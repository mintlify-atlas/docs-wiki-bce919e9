The ABP Framework keeps its own test suite next to its source code in two trees: `framework/test/` for unit and integration tests of every framework module, and `test/AbpPerfTest/` for a small ASP.NET Core throughput benchmark that compares an ABP-based app against a vanilla ASP.NET Core app. This page is the catalogue. It covers the supporting libraries first, then the integration test runners by domain, then the standalone benchmark.

All references are absolute paths under `/home/daytona/repos/abpframework/abp/`. Every `*.Tests` project in `framework/test/` follows the same csproj convention: it imports `..\..\..\common.test.props` (which centralises `Microsoft.NET.Test.Sdk`, `xunit`, `Shouldly`, `NSubstitute`), declares a `*.abppkg`, and lives in a folder named after the module under test.

## Supporting libraries

Three projects under `framework/test/` are libraries — they are referenced by other test projects rather than executed as runners.

### `framework/test/AbpTestBase/`

The shared Shouldly extensions and logger contracts for the entire suite. The csproj at `framework/test/AbpTestBase/AbpTestBase.csproj` references `Volo.Abp.TestBase` plus `Microsoft.NET.Test.Sdk`, `NSubstitute`, `Shouldly`, `xunit`, `xunit.extensibility.execution`, and `xunit.runner.visualstudio` — it carries the runner packages so they can be transitively pulled by every test project that references it.

The two important files are `framework/test/AbpTestBase/Microsoft/Extensions/DependencyInjection/ServiceCollectionShouldlyExtensions.cs`, which adds `services.ShouldContainTransient(typeof(IFoo), typeof(Foo))`, `ShouldContainScoped`, `ShouldContainSingleton`, `ShouldContainTransientImplementationFactory`, and `ShouldNotContainService` — used pervasively when a `ConfigureServices` is the unit under test — and `framework/test/AbpTestBase/Volo/Abp/TestBase/Logging/ICanLogOnObject.cs`, which is a tiny `interface ICanLogOnObject { List<string> Logs { get; } }` used by tests that need a fake log sink as part of their service graph.

### `framework/test/Volo.Abp.TestApp/`

The synthetic domain that data-access tests share. It defines `Person`, `City`, `District`, `EntityWithIntPk`, `AppEntityWithNavigations`, repositories (`IPersonRepository`, `ICityRepository`), DTOs, and application services. EF Core tests, Mongo tests, Dapper tests, memory-db tests, and specification tests all bind to this domain. The convention lets the framework write a single set of repository contracts and verify every backing store satisfies them. Browse `framework/test/Volo.Abp.TestApp/Volo/Abp/TestApp/` for the entity and DTO files.

### `framework/test/Volo.Abp.TestApp.Tests/`

Wires the `Volo.Abp.TestApp` domain into a runnable `*.Tests` project — it is the in-memory baseline that proves the synthetic domain compiles and works against `IRepository<T, TKey>` before any concrete backing store gets tested. Most data-access test projects don't run this directly; they reference `Volo.Abp.TestApp` and write their own runner against their own provider.

## Sample applications

Two folders are end-to-end samples used as smoke-test targets rather than as runners.

### `framework/test/SimpleConsoleDemo/`

The smallest possible ABP app. Its `Program.cs` at `framework/test/SimpleConsoleDemo/Program.cs` builds an `AbpApplicationFactory.Create<MyConsoleModule>(options => { options.Configuration.CommandLineArgs = args; options.UseAutofac(); })`, initialises it, resolves `IConfiguration` to check `AppSettingKey1`, then enumerates `IMessageWriter` services and writes each one's message. The whole thing is a single file with three types: `MyConsoleModule`, `ConsoleMessageWriter : IMessageWriter, ITransientDependency`, and `MessageSource : ITransientDependency`. The csproj at `framework/test/SimpleConsoleDemo/SimpleConsoleDemo.csproj` is `<OutputType>Exe</OutputType>`, references `Volo.Abp.Autofac`, and copies `appsettings.json` to the output directory. It is the sanity check that the module + Autofac + configuration pipeline still composes from scratch.

### `framework/test/Volo.Abp.AspNetCore.Mvc.PlugIn/`

Not a runner — a satellite assembly loaded as an MVC plug-in by `Volo.Abp.AspNetCore.Mvc.Tests`. `framework/test/Volo.Abp.AspNetCore.Mvc.PlugIn/MyPlungInModule.cs` declares `[DependsOn(typeof(AbpAspNetCoreMvcUiThemeSharedModule))]` and uses `PreConfigure<IMvcBuilder>` to add itself as both an `AssemblyPart` and a `CompiledRazorAssemblyPart`. The folder also contains `Volo/Abp/AspNetCore/Mvc/Index.cshtml` and `Index.cshtml.cs` — the actual razor view the parent test discovers at runtime. This project exists to prove that ABP's plug-in loading wires razor compilation correctly when an assembly is added after the host has started.

## Pattern summary

The framework groups its tests by base class as follows:

| Base | Test projects |
|---|---|
| No base (plain xUnit) | `Volo.Abp.AspNetCore.Authentication.OAuth.Tests`, `Volo.Abp.AspNetCore.Components.Web.Theming.Tests`, most of `Volo.Abp.Core.Tests`, primitive-only tests in many other folders. |
| `AbpIntegratedTest<TModule>` (sync) | `Volo.Abp.AI.Tests`, `Volo.Abp.AspNetCore.Mvc.Client.Tests`, `Volo.Abp.AutoMapper.Tests`, `Volo.Abp.Caching.Tests`, `Volo.Abp.EventBus.Tests`, `Volo.Abp.MultiTenancy.Tests`, `Volo.Abp.Settings.Tests`, `Volo.Abp.Uow.Tests` and most non-HTTP integration test projects. |
| `AbpAsyncIntegratedTest<TModule>` (async) | A handful of projects that need async module configuration — typically blob and data-provider variants that warm up a connection. |
| `AbpAspNetCoreTestBase<Program>` | Every `framework/test/Volo.Abp.AspNetCore.*.Tests/` HTTP runner except `Volo.Abp.AspNetCore.Mvc.Client.Tests` (which is integration only) and the two pure-unit ones above. |

Within each group, the convention is one `*TestModule` file per project, one `*TestBase` file per project, and one `Program.cs` per project (for HTTP runners) that calls `RunAbpModuleAsync<TStartupModule>` and ends with `public partial class Program {}`.

## `test/AbpPerfTest/` — the throughput benchmark

The performance suite at `/home/daytona/repos/abpframework/abp/test/AbpPerfTest/` is *not* an xUnit project. It is two side-by-side ASP.NET Core apps driven by a JMeter test plan, with results materialised as HTML and JSON. The folder layout:

```
test/AbpPerfTest/
├── AbpPerfTest.WithAbp/         # An ABP app with one Book resource
├── AbpPerfTest.WithoutAbp/      # A vanilla ASP.NET Core app with the same resource
├── _jmeter/
│   ├── SimpleTestPlan.jmx
│   └── SimpleTestPlanWithoutAbp.jmx
├── _results/
│   ├── WithAbp/    {WithAbp.jtl, index.html, statistics.json}
│   └── WithoutAbp/ {WithoutAbp.jtl, index.html, statistics.json}
└── AbpPerfTest.slnx
```

### `test/AbpPerfTest/AbpPerfTest.WithAbp/`

A full ABP application. Its `Program.cs` at `test/AbpPerfTest/AbpPerfTest.WithAbp/Program.cs`:

```csharp
var builder = WebApplication.CreateBuilder(args);
builder.Host.UseAutofac();
await builder.AddApplicationAsync<AppModule>();
var app = builder.Build();
await app.InitializeApplicationAsync();
await app.RunAsync();
```

`AppModule` lives at `test/AbpPerfTest/AbpPerfTest.WithAbp/AppModule.cs`. The HTTP surface is a single `BookController` at `test/AbpPerfTest/AbpPerfTest.WithAbp/Controllers/BookController.cs` with `BookDto` / `CreateUpdateBookDto` DTOs, a `Book` entity, and an EF Core DbContext. Migrations under `test/AbpPerfTest/AbpPerfTest.WithAbp/Migrations/` (`20231231021048_Added_Books.cs`) create the SQL schema.

### `test/AbpPerfTest/AbpPerfTest.WithoutAbp/`

The vanilla mirror. `test/AbpPerfTest/AbpPerfTest.WithoutAbp/Program.cs`:

```csharp
var builder = WebApplication.CreateBuilder(args);

builder.Services.AddMvc();
builder.Services.AddDbContext<BookDbContext>(options =>
{
    options.UseSqlServer(builder.Configuration.GetConnectionString("Default"));
});

var app = builder.Build();
if (builder.Environment.IsDevelopment()) app.UseDeveloperExceptionPage();
app.UseRouting();
app.UseEndpoints(endpoints => endpoints.MapControllers());
app.Run();
```

No `AppModule`, no Autofac, no `InitializeApplicationAsync` — straight MVC over EF Core. Controllers, DTOs, entity, DbContext, and migrations mirror the ABP project's layout exactly so the workload is identical.

### Running the benchmark

The `.jmx` files under `_jmeter/` drive both apps with the same scenario (the framework runs 160,000 requests per app — 80,000 reads, 80,000 writes — see `_results/WithAbp/statistics.json`). Results are committed back into `_results/<flavour>/` as `*.jtl` JMeter raw logs, an `index.html` dashboard, and a `statistics.json` summary.

A sample summary entry from `test/AbpPerfTest/_results/WithAbp/statistics.json` (160,000 total samples, 8,074 req/sec total throughput, 47 ms mean response time across reads + writes) lets the framework spot regression when a module change degrades the hot path.

## How to add a new test project

To verify a new framework module, copy the canonical layout from any `framework/test/Volo.Abp.<Module>.Tests/` directory: a csproj that imports `..\..\..\common.test.props` and references the source project plus `Volo.Abp.TestBase`, an `Abp<Module>TestModule.cs` that depends on `AbpTestBaseModule` and the module under test, a `<Subject>_Tests.cs` file deriving from `AbpIntegratedTest<Abp<Module>TestModule>`, and `[Fact]` methods that resolve through `GetRequiredService<T>()` and assert with Shouldly.

For an HTTP-pipelined module the layout adds three more files: a `Program.cs` that calls `await builder.RunAbpModuleAsync<Abp<Module>TestModule>();` and declares `public partial class Program {}`, an `Abp<Module>TestBase.cs` that says `public abstract class Abp<Module>TestBase : AbpAspNetCoreTestBase<Program> {}`, and concrete test classes deriving from that base.

The supporting libraries in `framework/test/AbpTestBase/` and `framework/test/Volo.Abp.TestApp/` are referenced from your csproj when you need their Shouldly extensions or shared synthetic domain — they pull in the runner packages transitively so you do not need to declare xUnit yourself.
