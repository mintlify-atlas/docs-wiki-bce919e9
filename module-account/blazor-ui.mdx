The ABP Framework Account module ships two near-identical Blazor packages: `modules/account/src/Volo.Abp.Account.Blazor/` (the Blazorise-themed default) and `modules/account/src/Volo.Abp.Account.Blazor.MudBlazor/`. Each one provides a single Razor component ‚Äî `AccountManage.razor` ‚Äî plus the menu contributor and the AutoMapper-style Mapperly profile that translates between `ProfileDto` and the local `PersonalInfoModel`. Authentication itself (login, register, password reset) is **not** part of the Blazor packages: those flows happen on the auth-server host through the OpenIddict or IdentityServer redirect chain.

## Package layout

Both packages have the same five files:

```
Volo.Abp.Account.Blazor/
  AbpAccountBlazorModule.cs
  AbpAccountBlazorMappers.cs
  AbpAccountBlazorUserMenuContributor.cs
  AbpAccountComponentBase.cs
  _Imports.razor
  Pages/Account/AccountManage.razor
  Pages/Account/AccountManage.razor.cs
```

The MudBlazor twin is at `Volo.Abp.Account.Blazor.MudBlazor/` with the same shape ‚Äî just a different module class name and a MudBlazor-flavored `.razor`. Both ship to NuGet as theme-specific assemblies and a Blazor application typically references exactly one.

## Module wiring (Blazorise)

`Volo.Abp.Account.Blazor/AbpAccountBlazorModule.cs`:

```csharp
[DependsOn(
    typeof(AbpAspNetCoreComponentsWebThemingModule),
    typeof(AbpMapperlyModule),
    typeof(AbpAccountApplicationContractsModule)
    )]
public class AbpAccountBlazorModule : AbpModule
{
    public override void ConfigureServices(ServiceConfigurationContext context)
    {
        context.Services.AddMapperlyObjectMapper<AbpAccountBlazorModule>();

        Configure<AbpNavigationOptions>(options =>
        {
            options.MenuContributors.Add(new AbpAccountBlazorUserMenuContributor());
        });

        Configure<AbpRouterOptions>(options =>
        {
            options.AdditionalAssemblies.Add(typeof(AbpAccountBlazorModule).Assembly);
        });
    }

    public override void PostConfigureServices(ServiceConfigurationContext context)
    {
        OneTimeRunner.Run(() =>
        {
            ModuleExtensionConfigurationHelper
                .ApplyEntityConfigurationToUi(
                    IdentityModuleExtensionConsts.ModuleName,
                    IdentityModuleExtensionConsts.EntityNames.User,
                    editFormTypes: new[] { typeof(PersonalInfoModel) }
                );
        });
    }
}
```

Three things happen here:

1. **The Mapperly mapper is registered** so `ObjectMapper.Map<ProfileDto, PersonalInfoModel>` works inside the component.
2. **The menu contributor is added** to `AbpNavigationOptions`.
3. **The assembly is added** to `AbpRouterOptions.AdditionalAssemblies` so the Blazor router discovers `AccountManage.razor`'s `@page "/account/manage-profile"` directive.
4. **`PostConfigureServices`** wires `PersonalInfoModel` into the Identity module's `ObjectExtensionManager` so any extra property defined on `IdentityUser` automatically appears on the manage form.

The dependency on `AbpAccountApplicationContractsModule` ‚Äî not the application package ‚Äî is intentional. Blazor WebAssembly hosts call the application services through the HTTP API client (`Volo.Abp.Account.HttpApi.Client`), so they only need the contracts. Blazor Server hosts can use either the in-process service or the client proxy depending on module composition.

## Module wiring (MudBlazor)

`Volo.Abp.Account.Blazor.MudBlazor/AbpAccountBlazorMudBlazorModule.cs` is a near-identical copy:

```csharp
[DependsOn(
    typeof(AbpAspNetCoreComponentsWebThemingMudBlazorModule),
    typeof(AbpMapperlyModule),
    typeof(AbpAccountApplicationContractsModule)
    )]
public class AbpAccountBlazorMudBlazorModule : AbpModule
{
    // ConfigureServices and PostConfigureServices identical to Blazorise version
}
```

The only difference is the theming dependency: `AbpAspNetCoreComponentsWebThemingMudBlazorModule` instead of `AbpAspNetCoreComponentsWebThemingModule`. Both contribute an `AbpAccountBlazorUserMenuContributor`, but each one is a different concrete type in its own namespace; the menu identifier `"Account.Manage"` is shared, so re-registering it would be a duplicate.

## `AccountManage.razor` (Blazorise)

`Volo.Abp.Account.Blazor/Pages/Account/AccountManage.razor` is a tabbed component routed at `/account/manage-profile` and `/{culture}/account/manage-profile`:

```razor
@page "/account/manage-profile"
@page "/{culture}/account/manage-profile"
@inject AbpBlazorMessageLocalizerHelper<AccountResource> LH
@inherits AbpAccountComponentBase

<Row>
    <Column ColumnSize="ColumnSize.Is12">
        <Tabs @bind-SelectedTab="@SelectedTab" TabPosition="TabPosition.Start" Pills="true">
            <Items>
                <Tab Name="Password">@L["ProfileTab:Password"]</Tab>
                <Tab Name="PersonalInfo">@L["ProfileTab:PersonalInfo"]</Tab>
            </Items>
            <Content>
                <TabPanel Name="Password"> ... </TabPanel>
                <TabPanel Name="PersonalInfo"> ... </TabPanel>
            </Content>
        </Tabs>
    </Column>
</Row>
```

The Blazorise version uses `Row`, `Column`, `Tabs`, `Tab`, `TabPanel`, `Field`, `FieldLabel`, `TextInput`, `EditForm`, and `SubmitButton` ‚Äî all Blazorise components. The Password panel renders an `EditForm` bound to `ChangePasswordModel`; the Personal Info panel binds to `PersonalInfoModel` and iterates `ObjectExtensionManager.Instance.GetProperties<PersonalInfoModel>()` to render any extension properties marked `AllowUserToEdit`:

```razor
@foreach (var propertyInfo in ObjectExtensionManager.Instance.GetProperties<PersonalInfoModel>())
{
    var isAllowed = propertyInfo.Configuration.GetOrDefault(
        IdentityModuleExtensionConsts.ConfigurationNames.AllowUserToEdit);

    if (isAllowed == null || !isAllowed.Equals(true)) continue;

    if (propertyInfo.IsEnum())
    {
        <SelectExtensionProperty PropertyInfo="@propertyInfo" Entity="@PersonalInfoModel"
            TEntity="PersonalInfoModel" TResourceType="AccountResource" LH="@LH" />
    }
    else if (!propertyInfo.Lookup.Url.IsNullOrEmpty())
    {
        <LookupExtensionProperty PropertyInfo="@propertyInfo" Entity="@PersonalInfoModel"
            TEntity="PersonalInfoModel" TResourceType="AccountResource" LH="@LH" />
    }
    else
    {
        var inputType = propertyInfo.GetInputType();
        __builder.OpenComponent(0, inputType.MakeGenericType(
            new[] { typeof(PersonalInfoModel), typeof(AccountResource) }));
        __builder.AddAttribute(1, "PropertyInfo", propertyInfo);
        __builder.AddAttribute(2, "Entity", PersonalInfoModel);
        __builder.AddAttribute(3, "LH", LH);
        __builder.CloseComponent();
    }
}
```

The fallback branch uses the low-level `__builder` API to instantiate a generic `<ExtensionProperty<TEntity, TResource>>` component because the type only becomes known at runtime via `propertyInfo.GetInputType()`. There is a `//TODO: Move this logic to 'ExtensionProperties' component.` note in the source ‚Äî the inline implementation predates the centralized extension-rendering component.

## Code-behind: `AccountManage.razor.cs`

`Volo.Abp.Account.Blazor/Pages/Account/AccountManage.razor.cs` carries the state and the handlers:

```csharp
public partial class AccountManage
{
    [Parameter] public string? Culture { get; set; }

    [Inject] protected IProfileAppService ProfileAppService { get; set; }
    [Inject] protected IUiMessageService UiMessageService { get; set; }

    protected string SelectedTab = "Password";
    protected ChangePasswordModel ChangePasswordModel;
    protected PersonalInfoModel PersonalInfoModel;

    protected override async Task OnInitializedAsync()
    {
        await GetUserInformations();
    }

    protected async Task GetUserInformations()
    {
        var user = await ProfileAppService.GetAsync();

        ChangePasswordModel = new ChangePasswordModel
        {
            HideOldPasswordInput = !user.HasPassword
        };

        PersonalInfoModel = ObjectMapper.Map<ProfileDto, PersonalInfoModel>(user);
    }

    protected async Task ChangePasswordAsync()
    {
        if (string.IsNullOrWhiteSpace(ChangePasswordModel.CurrentPassword)) return;

        if (ChangePasswordModel.NewPassword != ChangePasswordModel.NewPasswordConfirm)
        {
            await UiMessageService.Warn(L["NewPasswordConfirmFailed"]); return;
        }

        if (ChangePasswordModel.CurrentPassword == ChangePasswordModel.NewPassword)
        {
            await UiMessageService.Warn(L["NewPasswordSameAsOld"]); return;
        }

        await ProfileAppService.ChangePasswordAsync(new ChangePasswordInput
        {
            CurrentPassword = ChangePasswordModel.CurrentPassword,
            NewPassword = ChangePasswordModel.NewPassword
        });

        ChangePasswordModel.Clear();
        await UiMessageService.Success(L["PasswordChanged"]);
    }

    protected virtual async Task UpdatePersonalInfoAsync()
    {
        await ProfileAppService.UpdateAsync(
            ObjectMapper.Map<PersonalInfoModel, UpdateProfileDto>(PersonalInfoModel)
        );

        await UiMessageService.Success(L["PersonalSettingsSaved"]);
    }
}
```

Three implementation details worth highlighting:

- **`HideOldPasswordInput = !user.HasPassword`** mirrors the same logic from the application service ‚Äî external users have no password to confirm, so the field is omitted from the form. The view checks `@if (!ChangePasswordModel.HideOldPasswordInput)` before rendering the current-password input.
- **Client-side validation** for `NewPasswordSameAsOld` duplicates the server-side check inside `ChangePasswordInput.Validate`. Both run; the server-side one is the authoritative fail-safe.
- **`IUiMessageService`** comes from `Volo.Abp.AspNetCore.Components.Messages` and renders theme-appropriate toasts (Blazorise alerts here, `MudSnackbar` in the MudBlazor twin).

## `ChangePasswordModel` and `PersonalInfoModel`

Both models are declared in the same `.razor.cs` file:

```csharp
public class ChangePasswordModel
{
    public string CurrentPassword { get; set; }
    public string NewPassword { get; set; }
    public string NewPasswordConfirm { get; set; }
    public bool HideOldPasswordInput { get; set; }

    public void Clear() { CurrentPassword = ""; NewPassword = ""; NewPasswordConfirm = ""; }
}

public class PersonalInfoModel : ExtensibleObject
{
    public string UserName { get; set; }
    public string Email { get; set; }
    public string Name { get; set; }
    public string Surname { get; set; }
    public string PhoneNumber { get; set; }
    public bool PhoneNumberConfirmed { get; set; }
    public bool EmailConfirmed { get; set; }
    public string ConcurrencyStamp { get; set; }
}
```

`PersonalInfoModel` inheriting `ExtensibleObject` is what makes the extension-properties loop work ‚Äî extras are stored in the same `ExtraProperties` dictionary the Identity user has.

## `AbpAccountBlazorMappers`

`Volo.Abp.Account.Blazor/AbpAccountBlazorMappers.cs` defines the Mapperly profile (named `AbpAccountBlazorMappers` for symmetry with the AutoMapper-era convention, even though it is a Mapperly partial):

```csharp
[Mapper(RequiredMappingStrategy = RequiredMappingStrategy.Target)]
[MapExtraProperties]
public partial class ProfileDtoToPersonalInfoModelMapper : MapperBase<ProfileDto, PersonalInfoModel>
{
    [MapperIgnoreTarget(nameof(PersonalInfoModel.PhoneNumberConfirmed))]
    [MapperIgnoreTarget(nameof(PersonalInfoModel.EmailConfirmed))]
    public override partial PersonalInfoModel Map(ProfileDto source);

    [MapperIgnoreTarget(nameof(PersonalInfoModel.PhoneNumberConfirmed))]
    [MapperIgnoreTarget(nameof(PersonalInfoModel.EmailConfirmed))]
    public override partial void Map(ProfileDto source, PersonalInfoModel destination);
}

[Mapper(RequiredMappingStrategy = RequiredMappingStrategy.Target)]
[MapExtraProperties]
public partial class PersonalInfoModelToUpdateProfileDtoMapper : MapperBase<PersonalInfoModel, UpdateProfileDto>
{
    public override partial UpdateProfileDto Map(PersonalInfoModel source);
    public override partial void Map(PersonalInfoModel source, UpdateProfileDto destination);
}
```

`PhoneNumberConfirmed` and `EmailConfirmed` are deliberately ignored ‚Äî `ProfileDto` does not carry those booleans, and the form treats them as read-only display values populated through the (currently unused) extension hook. There is no `AccountBlazorAutoMapperProfile` in the source; the Mapperly equivalent above is the actual class registered with the framework.

## Menu contributor

`Volo.Abp.Account.Blazor/AbpAccountBlazorUserMenuContributor.cs`:

```csharp
public class AbpAccountBlazorUserMenuContributor : IMenuContributor
{
    public Task ConfigureMenuAsync(MenuConfigurationContext context)
    {
        if (context.Menu.Name != StandardMenus.User) return Task.CompletedTask;

        var accountResource = context.GetLocalizer<AccountResource>();

        context.Menu.AddItem(new ApplicationMenuItem("Account.Manage",
            accountResource["MyAccount"],
            url: "~/account/manage-profile",
            icon: "fa fa-cog"));

        return Task.CompletedTask;
    }
}
```

Compared to the MVC `AbpAccountUserMenuContributor`, the Blazor contributor only adds the "My Account" link. Logout is intentionally **not** added here ‚Äî Blazor hosts run the auth-server logout flow through a redirect to the host's `/Account/Logout`, which is provided by `Volo.Abp.Account.Web` and only renders on the host page, not inside Blazor.

## `AbpAccountComponentBase`

`Volo.Abp.Account.Blazor/AbpAccountComponentBase.cs` is the base class every Account component inherits:

```csharp
public abstract class AbpAccountComponentBase : AbpComponentBase
{
    protected AbpAccountComponentBase()
    {
        LocalizationResource = typeof(AccountResource);
        ObjectMapperContext = typeof(AbpAccountBlazorModule);
    }
}
```

Setting `LocalizationResource` means every `L["..."]` call inside the component resolves against `AccountResource` without an explicit injection. Setting `ObjectMapperContext` means `ObjectMapper.Map<...>(...)` resolves to the Mapperly mapper registered in `AbpAccountBlazorModule`. The MudBlazor twin has its own `AbpAccountComponentBase` in `Volo.Abp.Account.Blazor.MudBlazor/AbpAccountComponentBase.cs` with `ObjectMapperContext = typeof(AbpAccountBlazorMudBlazorModule)`.

## `_Imports.razor`

`Volo.Abp.Account.Blazor/_Imports.razor` (and its MudBlazor sibling) brings in the Blazorise/MudBlazor namespaces, the ABP Account namespaces, and the ABP component infrastructure. Without it, the strongly-typed components in the `.razor` file would not compile.

## What is *not* in the Blazor packages

- **No login/register/reset pages.** Authentication redirects to the auth-server host ‚Äî the Blazor app only sees the user *after* the cookie is set.
- **No permission/role management.** That lives in `Volo.Abp.Identity.Blazor` and `Volo.Abp.PermissionManagement.Blazor`.
- **No client-proxy registration.** That comes from `Volo.Abp.Account.HttpApi.Client` (see the [HTTP API page](/module-account/http-api)).

The Blazor packages are deliberately scoped to the self-service surface: one page, one menu item, one mapper.

<Note>
When you want to ship a custom profile tab in Blazor, the path is to write a new Razor component, add it as a third `<TabPanel>` inside a derived `AccountManage` component, and register that component as a service replacement via `[ExposeServices(typeof(AccountManage))]`. The `ProfileManagementPageOptions` / `IProfileManagementPageContributor` extension model from `Volo.Abp.Account.Web` is **MVC-only** ‚Äî it has no Blazor counterpart in the current source.
</Note>
