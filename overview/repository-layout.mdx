The ABP Framework repository is a single monorepo containing the framework runtime, eighteen feature modules, multiple solution templates, an npm workspace, ABP Studio source-code bundles, build automation, and the tooling that pushes packages to NuGet and the npm registry. This page is an annotated map of every top-level directory and the most useful sub-trees inside them. Every entry is grounded in real files so you can grep from the path and find the code immediately.

## Top-level directories

The repository root contains the following directories and key files:

| Path | Purpose |
| --- | --- |
| `framework/` | Core framework — all `Volo.Abp.*` runtime, DDD, data, HTTP, and UI packages. |
| `modules/` | 18 pre-built feature modules (identity, openiddict, tenant-management, cms-kit, …). |
| `templates/` | Solution scaffolds (`app`, `app-nolayers`, `module`, `console`, `maui`, `wpf`). |
| `npm/` | Yarn/Lerna workspace for Angular packages (`ng-packs/`) and MVC JS packs (`packs/`). |
| `studio/source-codes/` | `*.SourceCode` packages consumed by ABP Studio's "source-code module" feature. |
| `tools/` | Helper executables: `nuget/nuget.exe`, `github-changelog-generator/`, `localization-key-synchronizer/`. |
| `build/` | PowerShell scripts that build/test all solutions (`build-all.ps1`, `test-all.ps1`, `common.ps1`). |
| `deploy/` | Numbered release pipeline scripts (`1-fetch-and-build.ps1` through `8-download-release-zip.ps1`, `_run_all.ps1`). |
| `apiSpec/` | Manual API spec docs (e.g. `Microsoft_AspNetCore_Routing_AbpEndpointRouterOptions.md`). |
| `docs/` | Per-language documentation roots; configured by `docs/docs-langs.json`. |
| `test/` | Shared cross-cutting integration tests. |
| `source-code/` | Aggregated source-code solution used by ABP Studio's monorepo builds. |
| `abp_io/` | Sources for the abp.io site assets such as `abp_io/AbpIoLocalization/`. |
| `nupkg/` | Output and scripts (`pack.ps1`, `push_packages.ps1`) shared by deploy scripts. |
| `ai-rules/` | Coding/AI assistant rules consumed by the ABP AI tooling. |
| `Directory.Build.props` | Marks `*.Tests` / `*.TestBase` projects as `IsTestProject` and adds `coverlet.collector`. |
| `Directory.Packages.props` | Central package management — every `<PackageVersion>` entry that the whole repo shares. |
| `common.props` | Version (`<Version>10.4.0-rc.2</Version>`), `LangVersion`, NuGet metadata. |
| `common.test.props` | Common settings for test projects. |
| `configureawait.props` | Adds `ConfigureAwait.Fody` only in `Release` configuration. |
| `global.json` | Pins the .NET SDK (`10.0.100`) with `rollForward: latestFeature`. |
| `NuGet.Config` | Restricts package sources to `nuget.org` for restore reproducibility. |
| `NuGet.md` | Shared readme packed into every `.nupkg`. |
| `latest-versions.json` | Compatibility matrix used by the docs site and CLI. |
| `codecov.yml` | Codecov gating: `branch: dev`, 1 % threshold, coverage offsets allowed. |
| `delete-bin-obj.ps1` | Convenience script that cleans build outputs. |

<Note>
The repo uses **central package management**. `Directory.Packages.props` sets `<ManagePackageVersionsCentrally>true</ManagePackageVersionsCentrally>` and `<CentralPackageFloatingVersionsEnabled>true</CentralPackageFloatingVersionsEnabled>`, so individual `.csproj` files declare `<PackageReference Include="…" />` without versions and inherit the version from the props file.
</Note>

## `framework/` — the runtime

`framework/` ships the ABP Framework itself. It has the following structure:

```
framework/
├── NuGet.md
├── Volo.Abp.abpmdl          # ABP Studio module manifest (JSON)
├── Volo.Abp.abpsln          # ABP Studio solution manifest (JSON)
├── Volo.Abp.sln.DotSettings # ReSharper/Rider settings
├── Volo.Abp.slnx            # .NET 9 XML solution file (177 src + many test projects)
├── src/                     # 177 production packages
└── test/                    # cross-package integration tests
```

`framework/Volo.Abp.slnx` lists every project in the framework. Builds use this slnx; ABP Studio uses the `abpmdl` / `abpsln` pair, which group packages and solutions in JSON.

## `modules/` — pre-built feature modules

The 18 directories under `modules/` each ship one logical product feature. Every directory follows the same shape as `framework/`: an `*.abpmdl` + `*.abpsln` + `*.slnx` + `src/` + `test/` layout.

| Path | One-line description |
| --- | --- |
| `modules/account/` | Login/register/forgot-password UI and the account application service. |
| `modules/audit-logging/` | Audit-log persistence, EF/Mongo stores, Web/Blazor management UI. |
| `modules/background-jobs/` | Persistent job store and management UI (separate from the runtime in `framework/src/Volo.Abp.BackgroundJobs`). |
| `modules/basic-theme/` | The "Basic" Bootstrap theme used by the unlicensed templates. |
| `modules/blob-storing-database/` | Database-backed `BlobContainer` provider. |
| `modules/blogging/` | Lightweight blogging engine consumed by abp.io. |
| `modules/client-simulation/` | Tooling for stress-testing API surfaces. |
| `modules/cms-kit/` | Comments, ratings, reactions, tags, pages, menus, blogs. |
| `modules/docs/` | The documentation engine that powers abp.io/docs. |
| `modules/feature-management/` | UI and stores for `IFeatureProvider` overrides per edition/tenant. |
| `modules/identity/` | Users, roles, organisation units, claims, sessions, security logs. |
| `modules/identityserver/` | Legacy IdentityServer4 integration (kept for migration). |
| `modules/openiddict/` | OpenIddict-based authorisation server (current default). |
| `modules/permission-management/` | Permission grant storage and admin UI. |
| `modules/setting-management/` | Per-user/tenant/global setting store and admin UI. |
| `modules/tenant-management/` | Tenant CRUD, connection-string editor, hosting helpers. |
| `modules/users/` | Shared user abstractions and lookup services. |
| `modules/virtual-file-explorer/` | Browser for the in-memory virtual file system, useful during development. |

Each module lays out a five-tier pyramid. For example, `modules/identity/src/` contains `Volo.Abp.Identity.Domain.Shared/`, `Volo.Abp.Identity.Domain/`, `Volo.Abp.Identity.Application.Contracts/`, `Volo.Abp.Identity.Application/`, `Volo.Abp.Identity.HttpApi/`, `Volo.Abp.Identity.HttpApi.Client/`, `Volo.Abp.Identity.EntityFrameworkCore/`, `Volo.Abp.Identity.MongoDB/`, `Volo.Abp.Identity.Web/`, `Volo.Abp.Identity.AspNetCore/`, `Volo.Abp.Identity.Blazor/`, `Volo.Abp.Identity.Blazor.Server/`, `Volo.Abp.Identity.Blazor.WebAssembly/`, `Volo.Abp.Identity.Blazor.MudBlazor/`, `Volo.Abp.Identity.Blazor.MudBlazor.Server/`, `Volo.Abp.Identity.Blazor.MudBlazor.WebAssembly/`, `Volo.Abp.Identity.Installer/`, and `Volo.Abp.PermissionManagement.Domain.Identity/`.

## `templates/` — solution scaffolds

`templates/` ships every starter solution the CLI / ABP Studio can generate. Each template is a real, buildable solution that the build pipeline compiles to verify regressions.

```
templates/
├── Directory.Packages.props      # central package versions (mirrors repo root)
├── NuGet.Config
├── app/                          # Default layered application
│   ├── angular/                  # Angular SPA front-end
│   └── aspnet-core/              # Backend + MVC/Blazor/WebApp variants
├── app-nolayers/                 # Single-project (no DDD layering) app
│   ├── angular/
│   └── aspnet-core/
├── console/                      # Pure console host
│   ├── MyCompanyName.MyProjectName.slnx
│   ├── common.props
│   └── src/MyCompanyName.MyProjectName/
├── maui/                         # .NET MAUI client
├── module/                       # Reusable module template (mirrors modules/<name>)
│   ├── angular/
│   └── aspnet-core/{src, host, test, database}/
├── wpf/                          # WPF client
└── zip-templates.ps1             # Packs each template into a .zip for CLI consumption
```

## `build/` — local CI scripts

`build/` runs the same builds CI does. The scripts all share `build/common.ps1` for solution paths.

```powershell
# build/build-all.ps1
$full = $args[0]
. ".\common.ps1" $full
foreach ($solutionPath in $solutionPaths) {
    $solutionAbsPath = (Join-Path $rootFolder $solutionPath)
    Set-Location $solutionAbsPath
    dotnet build
    if (-Not $?) { Set-Location $rootFolder; exit $LASTEXITCODE }
}
Set-Location $rootFolder
```

`build/common.ps1` defines the solution list (`../framework`, `../modules/basic-theme`, `../modules/users`, …) and adds the templates and extras (`../templates/app/aspnet-core`, `../templates/console`, `../source-code`) when invoked with the `-f` flag. `build/test-all.ps1` repeats the loop calling `dotnet test --no-build --no-restore --collect:"XPlat Code Coverage"`. `build/build-all-release.ps1` is the Release-config variant used by the deploy pipeline.

## `deploy/` — release pipeline

`deploy/` contains the numbered release pipeline plus `deploy/_run_all.ps1` that orchestrates the eight steps in order:

<Steps>
  <Step title="1-fetch-and-build.ps1">
    Updates `common.props` `<Version>`, switches to the requested branch, runs `build/build-all-release.ps1`.
  </Step>
  <Step title="2-nuget-pack.ps1">
    Calls `nupkg/pack.ps1` to produce `.nupkg` files for every package.
  </Step>
  <Step title="3-nuget-push.ps1">
    Reads `nuget-api-key.txt` (or prompts) and pushes packages via `nupkg/push_packages.ps1`.
  </Step>
  <Step title="4-npm-publish-mvc.ps1">
    Publishes `npm/packs/*` via `npm/publish-mvc.ps1`.
  </Step>
  <Step title="5-npm-publish-angular.ps1">
    Publishes `npm/ng-packs/packages/*` via `npm/publish-ng.ps1`.
  </Step>
  <Step title="6-git-commit.ps1">
    Commits version bumps and tags the release.
  </Step>
  <Step title="7-publish-github-release.ps1">
    Creates the GitHub release using `deploy/new-github-release-function.psm1`.
  </Step>
  <Step title="8-download-release-zip.ps1">
    Downloads the GitHub-generated source zip for archival.
  </Step>
</Steps>

`deploy/plink.exe` ships with the scripts to allow SSH operations on Windows release agents.

## `tools/`

`tools/nuget/nuget.exe` is the classic `nuget.exe` used by deploy scripts that still need it (most use `dotnet pack` / `dotnet nuget push`). `tools/github-changelog-generator/` contains `generator.exe`, a `config.yaml`, a `src/` folder with the source, and a `README.md` describing how to regenerate the release notes. `tools/localization-key-synchronizer/` is a .NET tool (`LocalizationKeySynchronizer.exe`, `LocalizationKeySynchronizer.slnx`, `src/`, `README.md`) that compares localisation JSON resources between cultures and reports drift. `tools/smtp-prober-email-sender.exe` is a utility for verifying SMTP setups during deployment.

## `docs/`, `apiSpec/`, `source-code/`, `nupkg/`, `abp_io/`, `ai-rules/`

- `docs/docs-langs.json` declares which language codes the docs site renders.
- `apiSpec/` holds hand-curated API spec markdown like `apiSpec/Microsoft_AspNetCore_Routing_AbpEndpointRouterOptions.md`.
- `source-code/` is a meta-solution that aggregates framework + modules for ABP Studio's "open source code as a single solution" feature.
- `nupkg/` contains `common.ps1`, `pack.ps1`, and `push_packages.ps1` invoked by the `deploy/` scripts.
- `abp_io/` hosts content for the abp.io website (for example `abp_io/AbpIoLocalization/`).
- `ai-rules/` contains markdown rule sets consumed by the ABP AI tooling.

<Tip>
For navigation: think of the repo as **framework runtime (`framework/`) + opinionated extensions (`modules/`) + scaffolds that wire them up (`templates/`) + JS counterparts (`npm/`) + build/publish glue (`build/` & `deploy/`)**. Every directory above slots into one of those buckets.
</Tip>
