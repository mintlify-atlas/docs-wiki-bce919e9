The Basic Theme is the ABP Framework reference implementation of the `ITheme` contract — a deliberately minimal Bootstrap 5 / MudBlazor presentation layer that other commercial themes (LeptonX, Lepton) are expected to replace one-for-one. Everything in this page lives under `modules/basic-theme/src/`, which contains eleven projects covering MVC, Blazor Server, Blazor WebAssembly, and MudBlazor flavors, plus two NuGet installer packages.

## Package map

The eleven packages are mostly thin glue around the **Theme.Shared** infrastructure that lives outside this module. The MVC entry point is `Volo.Abp.AspNetCore.Mvc.UI.Theme.Basic`; the Blazor entry points are split into `Web` (shared component definitions), `Server` (interactive-server bundles), `WebAssembly` (router and identity glue) and a separate `WebAssembly.Bundling` package for the build-time CSS bundle contributors. The MudBlazor mirror has the same four-package shape. The two `*.Installer` projects are empty marker packages used by the ABP Suite installer to drop source files into a user's solution.

## MVC package: `Volo.Abp.AspNetCore.Mvc.UI.Theme.Basic`

The MVC theme is the heart of the module. Its `AbpAspNetCoreMvcUIBasicThemeModule.cs` declares `[DependsOn(typeof(AbpAspNetCoreMvcUiThemeSharedModule), typeof(AbpAspNetCoreMvcUiMultiTenancyModule))]`, registers itself as an application part in `PreConfigureServices` via `mvcBuilder.AddApplicationPartIfNotExists(...)`, and in `ConfigureServices` does four things: registers the theme on `AbpThemingOptions`, embeds its virtual files under `Volo.Abp.AspNetCore.Mvc.UI.Theme.Basic`, adds the `BasicThemeMainTopToolbarContributor`, and configures the global style + script bundles.

```csharp
// modules/basic-theme/src/Volo.Abp.AspNetCore.Mvc.UI.Theme.Basic/AbpAspNetCoreMvcUIBasicThemeModule.cs
Configure<AbpThemingOptions>(options =>
{
    options.Themes.Add<BasicTheme>();
    if (options.DefaultThemeName == null)
    {
        options.DefaultThemeName = BasicTheme.Name;
    }
});
```

### `BasicTheme : ITheme`

`modules/basic-theme/src/Volo.Abp.AspNetCore.Mvc.UI.Theme.Basic/BasicTheme.cs` is decorated with `[ThemeName(Name)]` where `Name = "Basic"`, marked `ITransientDependency`, and resolves the three standard layouts to embedded paths:

```csharp
// modules/basic-theme/src/Volo.Abp.AspNetCore.Mvc.UI.Theme.Basic/BasicTheme.cs
public virtual string GetLayout(string name, bool fallbackToDefault = true)
{
    switch (name)
    {
        case StandardLayouts.Application: return "~/Themes/Basic/Layouts/Application.cshtml";
        case StandardLayouts.Account:     return "~/Themes/Basic/Layouts/Account.cshtml";
        case StandardLayouts.Empty:       return "~/Themes/Basic/Layouts/Empty.cshtml";
        default: return fallbackToDefault ? "~/Themes/Basic/Layouts/Application.cshtml" : null;
    }
}
```

When a Razor Page or controller is rendered, `Volo.Abp.AspNetCore.Mvc.UI.Theming` resolves the current `ITheme` from `AbpThemingOptions` and calls `GetLayout(...)`. Pages set their layout name via `ViewBag` or the `AbpLayout` attribute; the layout filename is then injected before the view engine runs.

### `Application.cshtml` — the master layout

`modules/basic-theme/src/Volo.Abp.AspNetCore.Mvc.UI.Theme.Basic/Themes/Basic/Layouts/Application.cshtml` is the canonical MVC master layout. It injects `IBrandingProvider`, `IPageLayout`, and `IOptions<AbpThemingOptions>`; it renders `<title>` from `PageLayout.Content.Title`, applies an RTL `dir` attribute when `CultureHelper.IsRtl` is true, and pulls Bootstrap + theme assets through the `<abp-style-bundle name="@BasicThemeBundles.Styles.Global" />` and `<abp-script-bundle name="@BasicThemeBundles.Scripts.Global" />` tag helpers.

The layout is also a dense host for ABP's `LayoutHooks` extension points — `LayoutHooks.Head.First`, `LayoutHooks.Head.Last`, `LayoutHooks.Body.First`, `LayoutHooks.PageContent.First`, `LayoutHooks.PageContent.Last`, and `LayoutHooks.Body.Last` are all invoked through `Component.InvokeLayoutHookAsync(...)`. The body composes the navbar, page alerts, a content toolbar row, and the page body:

```cshtml
<!-- modules/basic-theme/src/Volo.Abp.AspNetCore.Mvc.UI.Theme.Basic/Themes/Basic/Layouts/Application.cshtml -->
@(await Component.InvokeAsync<MainNavbarViewComponent>())
<div class="@containerClass">
    @(await Component.InvokeAsync<PageAlertsViewComponent>())
    <div id="AbpContentToolbar"> ... </div>
    @RenderBody()
</div>
<abp-script-bundle name="@BasicThemeBundles.Scripts.Global" />
<script src="~/Abp/ApplicationLocalizationScript?cultureName=@CultureInfo.CurrentUICulture.Name"></script>
<script src="~/Abp/ApplicationConfigurationScript"></script>
<script src="~/Abp/ServiceProxyScript"></script>
```

`Account.cshtml` and `Empty.cshtml` (same folder) are stripped-down siblings — `Empty.cshtml` removes navbar/toolbar entirely, `Account.cshtml` swaps in a centered card container used by the Account module's login/register pages.

### Bundle definitions

`modules/basic-theme/src/Volo.Abp.AspNetCore.Mvc.UI.Theme.Basic/Bundling/BasicThemeBundles.cs` is the constants holder — both `Styles.Global` and `Scripts.Global` are simply the string `"Basic.Global"`. The bundles are wired in the module:

```csharp
// modules/basic-theme/src/Volo.Abp.AspNetCore.Mvc.UI.Theme.Basic/AbpAspNetCoreMvcUIBasicThemeModule.cs
options.StyleBundles.Add(BasicThemeBundles.Styles.Global, bundle => bundle
    .AddBaseBundles(StandardBundles.Styles.Global)
    .AddContributors(typeof(BasicThemeGlobalStyleContributor)));

options.ScriptBundles.Add(BasicThemeBundles.Scripts.Global, bundle => bundle
    .AddBaseBundles(StandardBundles.Scripts.Global)
    .AddContributors(typeof(BasicThemeGlobalScriptContributor)));
```

The contributors at `modules/basic-theme/src/Volo.Abp.AspNetCore.Mvc.UI.Theme.Basic/Bundling/BasicThemeGlobalStyleContributor.cs` and `BasicThemeGlobalScriptContributor.cs` add a single CSS / JS file each — they pin the theme's own `basic.css`/`basic.js` on top of the standard ABP base bundles inherited via `AddBaseBundles(StandardBundles.Styles.Global)`.

### View components

The Basic Theme exposes six view components, all under `modules/basic-theme/src/Volo.Abp.AspNetCore.Mvc.UI.Theme.Basic/Themes/Basic/Components/`:

<AccordionGroup>
  <Accordion title="MainNavbarViewComponent">
    `Themes/Basic/Components/MainNavbar/MainNavbarViewComponent.cs` renders `Default.cshtml` and is invoked from `Application.cshtml`. It is the visual container that composes the brand, menu, and toolbar slots into a single Bootstrap `<nav class="navbar">`.
  </Accordion>
  <Accordion title="MainNavbarBrandViewComponent">
    `Themes/Basic/Components/Brand/MainNavbarBrandViewComponent.cs` renders the app name/logo. The default view at `Themes/Basic/Components/Brand/Default.cshtml` calls `IBrandingProvider.AppName` so apps override by replacing the `IBrandingProvider`.
  </Accordion>
  <Accordion title="MainNavbarMenuViewComponent">
    `Themes/Basic/Components/Menu/MainNavbarMenuViewComponent.cs` resolves `IMenuManager`, calls `GetMainMenuAsync()`, and renders `Themes/Basic/Components/Menu/Default.cshtml` with each item delegated to `_MenuItem.cshtml` for recursive rendering.
  </Accordion>
  <Accordion title="MainNavbarToolbarViewComponent">
    `Themes/Basic/Components/Toolbar/MainNavbarToolbarViewComponent.cs` resolves `IToolbarManager`, calls `GetAsync(StandardToolbars.Main)`, and renders `Themes/Basic/Components/Toolbar/Default.cshtml`. Items are populated by `IToolbarContributor`s — including the theme's own `BasicThemeMainTopToolbarContributor`.
  </Accordion>
  <Accordion title="LanguageSwitchViewComponent / UserMenuViewComponent">
    `Themes/Basic/Components/Toolbar/LanguageSwitch/LanguageSwitchViewComponent.cs` and `Themes/Basic/Components/Toolbar/UserMenu/UserMenuViewComponent.cs` are the two concrete toolbar items added by the theme's contributor.
  </Accordion>
  <Accordion title="ContentTitleViewComponent / PageAlertsViewComponent">
    `Themes/Basic/Components/ContentTitle/ContentTitleViewComponent.cs` reads `IPageLayout.Content.Title` to render the breadcrumb header; `Themes/Basic/Components/PageAlerts/PageAlertsViewComponent.cs` flushes pending alert messages registered through `IAlertManager`.
  </Accordion>
</AccordionGroup>

### Menu rendering & `_MenuItem.cshtml`

`modules/basic-theme/src/Volo.Abp.AspNetCore.Mvc.UI.Theme.Basic/Themes/Basic/Components/Menu/Default.cshtml` iterates over the `ApplicationMenu.Items` list. Each `ApplicationMenuItem` is then rendered by `Themes/Basic/Components/Menu/_MenuItem.cshtml`, which:

- escapes `Model.ElementId` and `Model.CssClass` into raw attributes,
- resolves the URL via `Url.IsLocalUrl(Model.Url) ? Url.Content(Model.Url.EnsureStartsWith('~')) : Model.Url`,
- invokes a custom view component if `Model.GetComponentTypeOrDefault()` returned a non-null `ViewComponent` subtype,
- branches on `Model.IsLeaf` to emit either a flat `<a class="dropdown-item">` link or a nested `<ul class="dropdown-menu">` collapse.

This is the file you'd patch to add badges, icons-per-item, or per-item permission tooltips — and the file commercial themes most often replace wholesale.

### Toolbar contributor

`modules/basic-theme/src/Volo.Abp.AspNetCore.Mvc.UI.Theme.Basic/Toolbars/BasicThemeMainTopToolbarContributor.cs` is conditional in two ways: it only contributes when `context.Toolbar.Name == StandardToolbars.Main` and when `context.Theme is BasicTheme`. That second check is critical — it prevents the contributor from polluting LeptonX or custom themes that also chain on top of Theme.Shared:

```csharp
// modules/basic-theme/src/Volo.Abp.AspNetCore.Mvc.UI.Theme.Basic/Toolbars/BasicThemeMainTopToolbarContributor.cs
var languages = await languageProvider.GetLanguagesAsync();
if (languages.Count > 1)
{
    context.Toolbar.Items.Add(new ToolbarItem(typeof(LanguageSwitchViewComponent)));
}

if (context.ServiceProvider.GetRequiredService<ICurrentUser>().IsAuthenticated)
{
    context.Toolbar.Items.Add(new ToolbarItem(typeof(UserMenuViewComponent)));
}
```

## Blazor Web package: `Volo.Abp.AspNetCore.Components.Web.BasicTheme`

This is the *shared* Blazor surface — the components here are referenced by both Blazor Server and WebAssembly hosts. `modules/basic-theme/src/Volo.Abp.AspNetCore.Components.Web.BasicTheme/AbpAspNetCoreComponentsWebBasicThemeModule.cs` depends on `AbpAspNetCoreComponentsWebThemingModule` and registers the Blazor `BasicTheme` class against `AbpThemingOptions` exactly the same way the MVC variant does.

`modules/basic-theme/src/Volo.Abp.AspNetCore.Components.Web.BasicTheme/BasicTheme.cs` implements `ITheme` from `Volo.Abp.AspNetCore.Components.Web.Theming.Theming` — note the **type-based** layout resolution instead of the string paths used in MVC:

```csharp
// modules/basic-theme/src/Volo.Abp.AspNetCore.Components.Web.BasicTheme/BasicTheme.cs
public virtual Type GetLayout(string name, bool fallbackToDefault = true)
{
    switch (name)
    {
        case StandardLayouts.Application:
        case StandardLayouts.Account:
        case StandardLayouts.Empty:
            return typeof(MainLayout);
        default:
            return fallbackToDefault ? typeof(MainLayout) : typeof(NullLayout);
    }
}
```

All three standard layouts return the *same* `MainLayout` type — the Account and Empty variants are styled by CSS only.

The component files under `Themes/Basic/` are:

| File | Purpose |
| --- | --- |
| `App.razor` / `AppWithoutAuth.razor` | Root `<Router>` components for authenticated vs. anonymous hosts |
| `MainLayout.razor` + `.cs` | The `LayoutComponentBase` that frames every page |
| `NavMenu.razor` + `.cs` | Sidebar menu, builds from `IMenuManager.GetMainMenuAsync()` |
| `NavToolbar.razor` + `.cs` | Top-right toolbar, resolved through `IToolbarManager` |
| `Branding.razor` | Renders `IBrandingProvider` content |
| `FirstLevelNavMenuItem.razor` + `.cs` | First-level menu item renderer |
| `SecondLevelNavMenuItem.razor` + `.cs` | Recursively renders nested children |
| `NullLayout.razor` | Fall-through layout for unmatched names |
| `RedirectToLogin.razor` | Shared 401 redirect for both hosts |

## Blazor Server package: `Volo.Abp.AspNetCore.Components.Server.BasicTheme`

`modules/basic-theme/src/Volo.Abp.AspNetCore.Components.Server.BasicTheme/AbpAspNetCoreComponentsServerBasicThemeModule.cs` depends on the shared `Web.BasicTheme` module plus `AbpAspNetCoreComponentsServerThemingModule`. It adds the server-specific toolbar contributor and registers a global Blazor bundle:

```csharp
// modules/basic-theme/src/Volo.Abp.AspNetCore.Components.Server.BasicTheme/AbpAspNetCoreComponentsServerBasicThemeModule.cs
Configure<AbpToolbarOptions>(options =>
{
    options.Contributors.Add(new BasicThemeToolbarContributor());
});

Configure<AbpBundlingOptions>(options =>
{
    options.StyleBundles.Add(BlazorBasicThemeBundles.Styles.Global, bundle => bundle
        .AddBaseBundles(BlazorStandardBundles.Styles.Global)
        .AddContributors(typeof(BlazorBasicThemeStyleContributor)));
    options.ScriptBundles.Add(BlazorBasicThemeBundles.Scripts.Global, bundle => bundle
        .AddBaseBundles(BlazorStandardBundles.Scripts.Global)
        .AddContributors(typeof(BlazorBasicThemeScriptContributor)));
});
```

Notable Server-only files: `Themes/Basic/LoginDisplay.razor` and `LoginDisplay.razor.cs` — the server-side authentication header (it triggers a normal HTTP POST to `/Account/Logout` rather than calling an OIDC client) and `Themes/Basic/LanguageSwitch.razor`, a Blazor render of the same toolbar item rendered by the MVC `LanguageSwitchViewComponent`.

## Blazor WebAssembly package: `Volo.Abp.AspNetCore.Components.WebAssembly.BasicTheme`

`modules/basic-theme/src/Volo.Abp.AspNetCore.Components.WebAssembly.BasicTheme/AbpAspNetCoreComponentsWebAssemblyBasicThemeModule.cs` depends on the WASM-side bundling module, the shared `Web.BasicTheme`, `AbpAspNetCoreComponentsWebAssemblyThemingModule`, and the IdentityModel WebAssembly client. In `ConfigureServices` it:

1. Registers its own assembly with `AbpRouterOptions.AdditionalAssemblies` so the `<Router>` discovers `Pages/Authentication.razor` and the WASM-only `LoginDisplay`.
2. Adds the `BasicThemeToolbarContributor` for WASM (the WASM contributor differs from the Server one because login/logout flow goes through OIDC client-side).
3. If running in a Blazor *Web App* host (interactive WebAssembly under a Server host) — `context.Services.ExecutePreConfiguredActions<AbpAspNetCoreComponentsWebOptions>().IsBlazorWebApp` — it overrides `AuthenticationOptions.LoginUrl` and `LogoutUrl` to `"Account/Login"` / `"Account/Logout"` so the OIDC redirect points at the server's MVC endpoints.

`modules/basic-theme/src/Volo.Abp.AspNetCore.Components.WebAssembly.BasicTheme/Pages/Authentication.razor` handles the OIDC callback. `Themes/Basic/WebAssemblyRedirectToLogin.razor` is the WASM-flavored 401 redirect (it triggers `NavigationManager.NavigateToLogin(...)` rather than a server redirect).

### `WebAssembly.BasicTheme.Bundling`

`modules/basic-theme/src/Volo.Abp.AspNetCore.Components.WebAssembly.BasicTheme.Bundling/AbpAspNetCoreComponentsWebAssemblyBasicThemeBundlingModule.cs` is a *build-time-only* module used by the `ABP CLI` `bundle` command during `dotnet publish`. It depends on `AbpAspNetCoreComponentsWebAssemblyThemingBundlingModule` and patches the global WASM style bundle with `BasicThemeBundleStyleContributor`, which lives at `BasicThemeBundleStyleContributor.cs` in the same folder. This split lets WASM CSS be statically pre-bundled into a single hashed file at publish time, instead of fetched dynamically.

## MudBlazor packages

The MudBlazor variant duplicates the four-package shape with a separate theme name. `modules/basic-theme/src/Volo.Abp.AspNetCore.Components.Web.MudBlazorBasicTheme/MudBlazorBasicTheme.cs` registers itself with `Name = "MudBlazorBasic"` and returns `typeof(MainLayout)` for all three standard layouts — the same shape as the Bootstrap variant but resolving to `Themes/Basic/MainLayout.razor` *inside the MudBlazor assembly*. That layout uses `<MudThemeProvider>`, `<MudDialogProvider>`, and `<MudSnackbarProvider>` rather than Bootstrap navbars.

### MudBlazor module wiring

`modules/basic-theme/src/Volo.Abp.AspNetCore.Components.Web.MudBlazorBasicTheme/AbpAspNetCoreComponentsWebMudBlazorBasicThemeModule.cs` is the analog of the Web shared module — it depends on `AbpAspNetCoreComponentsWebThemingMudBlazorModule` and registers `MudBlazorBasicTheme` as the default. `modules/basic-theme/src/Volo.Abp.AspNetCore.Components.Server.MudBlazorBasicTheme/AbpAspNetCoreComponentsServerMudBlazorBasicThemeModule.cs` registers the `MudBlazorBasicThemeToolbarContributor` and configures `BlazorMudBlazorBasicThemeBundles.Styles.Global` / `Scripts.Global`. `modules/basic-theme/src/Volo.Abp.AspNetCore.Components.WebAssembly.MudBlazorBasicTheme/AbpAspNetCoreComponentsWebAssemblyMudBlazorBasicThemeModule.cs` again pushes the assembly onto `AbpRouterOptions.AdditionalAssemblies` and overrides `AuthenticationOptions` for Blazor Web App hosts. The bundling project `modules/basic-theme/src/Volo.Abp.AspNetCore.Components.WebAssembly.MudBlazorBasicTheme.Bundling/AbpAspNetCoreComponentsWebAssemblyMudBlazorBasicThemeBundlingModule.cs` extends `BlazorWebAssemblyMudBlazorStandardBundles.Styles.Global` with `WebAssemblyMudBlazorBasicThemeBundleStyleContributor`.

### MudBlazor component differences

The MudBlazor variant ships fewer Razor files because MudBlazor's own `<MudNavMenu>` and `<MudAppBar>` collapse a lot of structure. The notable files are:

- `modules/basic-theme/src/Volo.Abp.AspNetCore.Components.Web.MudBlazorBasicTheme/Themes/Basic/MainLayout.razor` — wraps the app in `<MudLayout>` with `<MudAppBar>` + `<MudDrawer>`.
- `Themes/Basic/NavMenu.razor` (+ `.cs`) — renders `<MudNavMenu>`/`<MudNavLink>` tree.
- `Themes/Basic/NavMenuItem.razor` — single recursive item template (replaces the two-level `FirstLevelNavMenuItem` / `SecondLevelNavMenuItem` split from the Bootstrap variant).
- `Themes/Basic/NavToolbar.razor` (+ `.cs`) — composes `<MudIconButton>`-based toolbar items.

## Installer packages

The two installer packages contain no runtime code. `modules/basic-theme/src/Volo.Abp.AspNetCore.Mvc.UI.Theme.Basic.Installer/InstallationNotes.md` and `modules/basic-theme/src/Volo.Abp.BasicTheme.Installer/InstallationNotes.md` carry the human-readable instructions shown by ABP Suite when a user adds the package to a solution. The latter also includes `FodyWeavers.xml` / `FodyWeavers.xsd`, indicating the umbrella installer uses Fody to weave its own resources at build time.

<Note>
Because the Basic Theme is the *default* theme, replacing it with LeptonX or a custom theme is a matter of replacing the module dependency in your `*.Web` or `*.Blazor` project's module class — the `[DependsOn]` graph then transitively swaps `BasicTheme.Name` for the new theme's name in `AbpThemingOptions.DefaultThemeName`.
</Note>

## End-to-end request flow (MVC)

<Steps>
  <Step title="Request hits a Razor Page">
    A Razor Page in the host app (for example `Pages/Index.cshtml`) is selected by ASP.NET Core routing. The page has no explicit `Layout = ...` because ABP injects it.
  </Step>
  <Step title="Theme resolution">
    `Volo.Abp.AspNetCore.Mvc.UI.Theming.AbpLayoutPageFilter` runs before the view executes. It reads `AbpThemingOptions.DefaultThemeName` (set to `"Basic"` by `AbpAspNetCoreMvcUIBasicThemeModule.ConfigureServices`) and resolves `BasicTheme` via DI.
  </Step>
  <Step title="Layout lookup">
    The filter calls `BasicTheme.GetLayout(StandardLayouts.Application)` which returns `"~/Themes/Basic/Layouts/Application.cshtml"`. That path is embedded as a virtual file by the `AddEmbedded<AbpAspNetCoreMvcUiBasicThemeModule>(...)` call.
  </Step>
  <Step title="Layout renders">
    `Application.cshtml` runs the bundle tag helpers, the `MainNavbar` view component, then your page body, then the script bundles + ABP application scripts.
  </Step>
  <Step title="Toolbar composes">
    Inside the navbar, `MainNavbarToolbarViewComponent` calls `IToolbarManager.GetAsync(StandardToolbars.Main)`. The manager iterates all registered `IToolbarContributor`s — `BasicThemeMainTopToolbarContributor` adds `LanguageSwitch` + `UserMenu` items conditionally; other modules (Account, Identity, AuditLogging) add their own.
  </Step>
</Steps>

## Customization patterns

The Basic Theme is meant to be *extended*, not patched in place. The supported extension points are:

- **Layout hooks** — `Volo.Abp.AspNetCore.Mvc.UI.Components.LayoutHook` registers components against `LayoutHooks.Body.First` etc. and they are invoked from `Application.cshtml` without changing the theme source.
- **Toolbar contributors** — add your own `IToolbarContributor` in your module's `ConfigureServices`; if you guard on `context.Theme is BasicTheme` you can target this theme only.
- **Bundle contributors** — register against `BasicThemeBundles.Styles.Global` / `BasicThemeBundles.Scripts.Global` to append CSS/JS without touching the theme.
- **Menu contributors** — implement `IMenuContributor` to add `ApplicationMenuItem`s; the theme's `MainNavbarMenuViewComponent` will pick them up.
- **Branding** — replace `IBrandingProvider` to change `AppName`, logo, and favicon without subclassing.
- **Page Alerts** — push messages via `IAlertManager`; they are rendered automatically by `PageAlertsViewComponent`.

For deeper changes, the recommended pattern is to copy the Razor views into your own `Themes/Basic/Layouts/Application.cshtml` path inside the host project — the `AddEmbedded` registration uses lowest-priority semantics in the virtual file system, so a host-level copy wins.
