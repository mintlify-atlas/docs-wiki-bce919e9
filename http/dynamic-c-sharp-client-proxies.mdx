## What "dynamic C# proxy" means in ABP Framework

When an ABP Framework client project wants to call a remote application service over HTTP, it does **not** hand-write an `HttpClient` per method. Instead it registers the service interface (any interface deriving from `IRemoteService`) and ABP installs a Castle DynamicProxy interceptor that turns every method call into an HTTP request to a remote host. The plumbing lives in `framework/src/Volo.Abp.Http.Client` and the entry point is `ServiceCollectionHttpClientProxyExtensions.AddHttpClientProxy` in `Microsoft/Extensions/DependencyInjection/ServiceCollectionHttpClientProxyExtensions.cs`.

This page walks through the runtime path: how the proxy is created, how it discovers the matching server-side action, how the request is built, and how the response is materialised.

<Info>
The companion package `Volo.Abp.Http.Client` ships two flavours of proxies: **dynamic** proxies (this page), which fetch `ApplicationApiDescriptionModel` at runtime from `/api/abp/api-definition`, and **static** proxies (generated by the `abp generate-proxy` CLI command and documented in [CLI generate-proxy](/cli/generate-proxy)), which are pre-built at compile time. Both share the `ClientProxyBase<TService>` base class.
</Info>

## Registering proxies: `AddHttpClientProxies`

`ServiceCollectionHttpClientProxyExtensions.AddHttpClientProxies(Assembly, string, bool, ApplicationServiceTypes)` enumerates every public, non-generic interface in the supplied assembly that implements `IRemoteService` and registers it as an HTTP proxy. The filter is `IsSuitableForClientProxying` in the same file:

```csharp
if (!type.IsInterface ||
    !type.IsPublic ||
    type.IsGenericType ||
    !typeof(IRemoteService).IsAssignableFrom(type))
{
    return false;
}
```

For each matching interface the extension calls `AddHttpClientProxy(Type, string, bool)`. That method does three things:

1. Calls `AddHttpClientFactory(services, remoteServiceConfigurationName)` to register a named `HttpClient` via `services.AddHttpClient(remoteServiceConfigurationName, …)` and applies the `AbpHttpClientBuilderOptions.ProxyClientActions`, `ProxyClientHandlerActions`, and `ProxyClientBuildActions` collected through `services.PreConfigure<AbpHttpClientBuilderOptions>(…)`.
2. Adds the service to `AbpHttpClientOptions.HttpClientProxies` (a `Dictionary<Type, HttpClientProxyConfig>`) using `new HttpClientProxyConfig(type, remoteServiceConfigurationName)`. `HttpClientProxyConfig` in `Volo/Abp/Http/Client/Proxying/HttpClientProxyConfig.cs` just records the service `Type` and the `RemoteServiceName`.
3. Registers the closed generic `DynamicHttpProxyInterceptor<TService>` as transient and creates a Castle proxy that uses the `AbpAsyncDeterminationInterceptor<ValidationInterceptor>` and `AbpAsyncDeterminationInterceptor<DynamicHttpProxyInterceptor<T>>` as its only interceptors.

The proxy itself is produced by a static `ProxyGenerator` instance:

```csharp
ProxyGeneratorInstance.CreateInterfaceProxyWithoutTarget(
    type,
    (IInterceptor)serviceProvider.GetRequiredService(validationInterceptorAdapterType),
    (IInterceptor)serviceProvider.GetRequiredService(interceptorAdapterType));
```

There is no target object — every call goes straight to the interceptor chain. The same registration also adds `IHttpClientProxy<T>` (implemented by `HttpClientProxy<TRemoteService>` in `Volo/Abp/Http/Client/DynamicProxying/HttpClientProxy.cs`) so consumers that need an *additional* proxy alongside the default registration can inject `IHttpClientProxy<IBookAppService>` instead of fighting DI ordering.

<Tip>
If you call `AddHttpClientProxies(assembly, asDefaultServices: false)` the framework still registers `IHttpClientProxy<T>` but skips the `services.AddTransient(type, …)` line, so the interface is *not* the default DI binding for `T`. This is how host-side modules can keep the real application service as the default and still expose a remote proxy for, say, integration tests.
</Tip>

## The interceptor: `DynamicHttpProxyInterceptor<TService>`

`Volo/Abp/Http/Client/DynamicProxying/DynamicHttpProxyInterceptor.cs` defines:

```csharp
public class DynamicHttpProxyInterceptor<TService>
    : AbpInterceptor, ITransientDependency
```

It depends on `DynamicHttpProxyInterceptorClientProxy<TService>` (a thin subclass of `ClientProxyBase<TService>`), `IOptions<AbpHttpClientOptions>`, `IProxyHttpClientFactory`, `IRemoteServiceConfigurationProvider`, and `IApiDescriptionFinder`. The `InterceptAsync(IAbpMethodInvocation)` override is short:

```csharp
var context = new ClientProxyRequestContext(
    await GetActionApiDescriptionModel(invocation),
    invocation.ArgumentsDictionary,
    typeof(TService));

if (invocation.Method.ReturnType.GenericTypeArguments.IsNullOrEmpty())
{
    await InterceptorClientProxy.CallRequestAsync(context);
}
else
{
    var returnType = invocation.Method.ReturnType.GenericTypeArguments[0];
    var result = (Task)CallRequestAsyncMethod
        .MakeGenericMethod(returnType)
        .Invoke(this, new object[] { context })!;

    invocation.ReturnValue = await GetResultAsync(result, returnType);
}
```

Two interesting details:

- `CallRequestAsyncMethod` is a cached `MethodInfo` captured in the static constructor via reflection (`GetMethods(BindingFlags.NonPublic | BindingFlags.Instance).First(m => m.Name == nameof(CallRequestAsync) && m.IsGenericMethodDefinition)`). At call time it is closed over the actual `T` extracted from `Task<T>`.
- If the return type is non-generic (`Task`), the interceptor calls the non-generic overload of `CallRequestAsync` which returns an `HttpContent`; the result is discarded but the HTTP request is awaited.

### Resolving the `ActionApiDescriptionModel`

`GetActionApiDescriptionModel` is the bridge between the C# method that was invoked and the server-side route metadata:

```csharp
var clientConfig = ClientOptions.HttpClientProxies.GetOrDefault(typeof(TService))
    ?? throw new AbpException($"Could not get DynamicHttpClientProxyConfig for {typeof(TService).FullName}.");
var remoteServiceConfig = await RemoteServiceConfigurationProvider
    .GetConfigurationOrDefaultAsync(clientConfig.RemoteServiceName);
var client = HttpClientFactory.Create(clientConfig.RemoteServiceName);

return await ApiDescriptionFinder.FindActionAsync(
    client,
    remoteServiceConfig.BaseUrl,
    typeof(TService),
    invocation.Method);
```

`IRemoteServiceConfigurationProvider` returns the `RemoteServiceConfiguration` registered in `AbpRemoteServiceOptions`. `IProxyHttpClientFactory` is implemented by `DefaultProxyHttpClientFactory` (`Volo/Abp/Http/Client/Proxying/DefaultProxyHttpClientFactory.cs`), which just delegates to `IHttpClientFactory.CreateClient(name)`.

## `IApiDescriptionFinder`: discovering the remote action

`ApiDescriptionFinder` in `Volo/Abp/Http/Client/DynamicProxying/ApiDescriptionFinder.cs` is the runtime equivalent of a Swagger schema reader. It performs a GET against `{baseUrl}/api/abp/api-definition`, deserialises the JSON into an `ApplicationApiDescriptionModel`, and caches it through `IApiDescriptionCache`:

```csharp
public virtual async Task<ApplicationApiDescriptionModel> GetApiDescriptionAsync(HttpClient client, string baseUrl)
{
    return await Cache.GetAsync(baseUrl, () => GetApiDescriptionFromServerAsync(client, baseUrl));
}
```

The HTTP fetch adds three headers in `AddHeaders`:

| Header | Source |
| --- | --- |
| `AbpCorrelationIdOptions.HttpHeaderName` | `ICorrelationIdProvider.Get()` |
| `TenantResolverConsts.DefaultTenantKey` | `ICurrentTenant.Id` (when present) |
| `Accept-Language` | `CultureInfo.CurrentUICulture.Name` |
| `X-Requested-With` | hard-coded `XMLHttpRequest` |

After the description is loaded, `FindActionAsync(client, baseUrl, serviceType, method)` walks `ApplicationApiDescriptionModel.Modules → Controllers → Actions` looking for a controller whose `Implements(serviceType)` returns true and an action whose `Name` matches `method.Name` *and* whose `ParametersOnMethod` count matches the C# parameter count. Type matching is delegated to `TypeMatches`:

```csharp
return actionParameter.Type.ToUpper()
    == TypeHelper.GetFullNameHandlingNullableAndGenerics(methodParameter.ParameterType).ToUpper();
```

If nothing matches the finder throws `AbpException($"Could not find remote action for method: {method} on the URL: {baseUrl}")`. The cache means this lookup typically costs one HTTP round-trip per `baseUrl` per process lifetime.

<Warning>
Because the comparison is case-insensitive on the **fully qualified type name**, you cannot freely use namespace-shadowing or differently-namespaced DTOs on client and server. Contract DTOs should live in a shared package — typically the `.Application.Contracts` project — which is exactly the convention the ABP application module template enforces.
</Warning>

## Sequence of an interceptor call

```mermaid
sequenceDiagram
    autonumber
    participant App as Client Code
    participant Proxy as Castle Proxy (IBookAppService)
    participant Validation as ValidationInterceptor
    participant Dyn as DynamicHttpProxyInterceptor&lt;T&gt;
    participant Finder as ApiDescriptionFinder
    participant Base as ClientProxyBase&lt;T&gt;
    participant Http as HttpClient
    participant Server as Remote Host

    App->>Proxy: await service.GetAsync(id)
    Proxy->>Validation: InterceptAsync(invocation)
    Validation->>Dyn: InterceptAsync(invocation)
    Dyn->>Finder: FindActionAsync(client, baseUrl, T, method)
    alt cache miss
        Finder->>Server: GET /api/abp/api-definition
        Server-->>Finder: ApplicationApiDescriptionModel JSON
    end
    Finder-->>Dyn: ActionApiDescriptionModel
    Dyn->>Base: CallRequestAsync<TResult>(ClientProxyRequestContext)
    Base->>Http: SendAsync(HttpRequestMessage)
    Http->>Server: GET /api/app/book/{id}
    Server-->>Http: 200 OK + JSON
    Http-->>Base: HttpResponseMessage
    Base-->>Dyn: TResult (deserialized)
    Dyn-->>App: invocation.ReturnValue
```

## `DynamicHttpProxyInterceptorClientProxy<TService>`

This class in `Volo/Abp/Http/Client/DynamicProxying/DynamicHttpProxyInterceptorClientProxy.cs` is the simplest possible subclass of `ClientProxyBase<TService>`:

```csharp
public class DynamicHttpProxyInterceptorClientProxy<TService> : ClientProxyBase<TService>
{
    public virtual async Task<T> CallRequestAsync<T>(ClientProxyRequestContext requestContext)
        => await base.RequestAsync<T>(requestContext);

    public virtual async Task<HttpContent> CallRequestAsync(ClientProxyRequestContext requestContext)
        => await base.RequestAsync(requestContext);
}
```

It exists purely to make `RequestAsync` callable from outside the proxy hierarchy — `ClientProxyBase` defines those methods as `protected virtual`.

## `ClientProxyBase<TService>`: the shared HTTP pipeline

`Volo/Abp/Http/Client/ClientProxying/ClientProxyBase.cs` is where the actual HTTP request is assembled. It uses `IAbpLazyServiceProvider` to pull in roughly fifteen services on demand (correlation ID, current tenant, JSON serialiser, authenticator, URL builder, payload builder, current API version, local event bus). The relevant excerpt for the dynamic path is `RequestAsync(ClientProxyRequestContext)`:

```csharp
var clientConfig = ClientOptions.Value.HttpClientProxies.GetOrDefault(requestContext.ServiceType)
    ?? throw new AbpException($"Could not get HttpClientProxyConfig for {requestContext.ServiceType.FullName}.");
var remoteServiceConfig = await RemoteServiceConfigurationProvider
    .GetConfigurationOrDefaultAsync(clientConfig.RemoteServiceName);

var client = HttpClientFactory.Create(clientConfig.RemoteServiceName);

var apiVersion = await GetApiVersionInfoAsync(requestContext);
var url = remoteServiceConfig.BaseUrl.EnsureEndsWith('/')
    + await GetUrlWithParametersAsync(requestContext, apiVersion);

var requestMessage = new HttpRequestMessage(requestContext.Action.GetHttpMethod(), url)
{
    Content = await ClientProxyRequestPayloadBuilder.BuildContentAsync(
        requestContext.Action, requestContext.Arguments, JsonSerializer, apiVersion)
};

AddHeaders(requestContext.Arguments, requestContext.Action, requestMessage, apiVersion);

if (requestContext.Action.AllowAnonymous != true)
{
    await ClientAuthenticator.Authenticate(
        new RemoteServiceHttpClientAuthenticateContext(
            client, requestMessage, remoteServiceConfig, clientConfig.RemoteServiceName));
}
```

The four cooperating helpers come from the same folder:

- **`ClientProxyUrlBuilder`** (not shown here) — turns the action's `Url` template plus path/query parameters into a relative URL and merges any `ApiVersionInfo` placeholders.
- **`ClientProxyRequestPayloadBuilder`** — produces an `HttpContent` for body parameters: JSON for normal DTOs, `MultipartFormDataContent` for file uploads (via `IObjectToFormData`), or `FormUrlEncodedContent` when the action's `SupportedRequestFormats` so dictates.
- **`IRemoteServiceHttpClientAuthenticator`** — the only line in this whole flow that touches credentials. The default `NullRemoteServiceHttpClientAuthenticator` is a no-op; modules like `Volo.Abp.Http.Client.IdentityModel` replace it with one that attaches a bearer token from `IAbpAccessTokenProvider`.
- **`AddHeaders`** — copies correlation id, tenant id (when set), current culture, current timezone (from `ICurrentTimezoneProvider`), and any `[Header]`-bound parameters from `requestContext.Arguments` into the outbound message.

After `SendAsync` returns, `ClientProxyBase` reads the response. For `Task<T>` the body is buffered into a string and run through `IJsonSerializer.Deserialize<T>`. For `Task<IRemoteStreamContent>` it instead constructs a `RemoteStreamContent` that holds the live `Stream`, the parsed `ContentDisposition.FileName`, and the `Content-Type` header. For `Task<string>` it returns the raw string. Errors translate into `AbpRemoteCallException` via `try/catch` around the `SendAsync` call.

## `HttpClientProxyConfig` and `AbpHttpClientOptions`

`AbpHttpClientOptions` (`Volo/Abp/Http/Client/AbpHttpClientOptions.cs`) holds two dictionaries:

```csharp
public Dictionary<Type, HttpClientProxyConfig> HttpClientProxies { get; set; }

public Dictionary<string,
    List<Action<HttpClientProxyConfig, ClientProxyRequestContext, HttpClient>>>
    ProxyHttpClientPreSendActions { get; }
```

Module authors mutate the first via `AddHttpClientProxy` (above) and the second via the `AddPreSendAction(remoteServiceName, …)` fluent helper. The pre-send actions run inside `ClientProxyBase.RequestAsync` immediately before `client.SendAsync`, which is the right place to set per-call headers or to override the URL on a per-tenant basis.

`AbpHttpClientBuilderOptions` (`Volo/Abp/Http/Client/AbpHttpClientBuilderOptions.cs`) is a **pre-configure-only** options class. Modules pre-configure `ProxyClientActions`, `ProxyClientHandlerActions`, and `ProxyClientBuildActions` so that they run inside the `AddHttpClient(name, …)` factory builder once per remote service. The `ConfiguredProxyClients` `HashSet` guards against double-registration when multiple modules call `AddHttpClientProxies` for the same remote service name.

## `IRemoteServiceHttpClientAuthenticator`

This one-method interface (`Volo/Abp/Http/Client/Authentication/IRemoteServiceHttpClientAuthenticator.cs`) is the seam where authentication plugs into the proxy pipeline:

```csharp
public interface IRemoteServiceHttpClientAuthenticator
{
    Task Authenticate(RemoteServiceHttpClientAuthenticateContext context);
}
```

`RemoteServiceHttpClientAuthenticateContext` exposes the `HttpClient`, the `HttpRequestMessage`, the `RemoteServiceConfiguration`, and the configuration name. The default `NullRemoteServiceHttpClientAuthenticator` does nothing; in real applications the `Volo.Abp.Http.Client.IdentityModel` package replaces it with an implementation that calls `IAbpAccessTokenProvider.GetTokenAsync()` and sets the `Authorization: Bearer …` header.

## API versioning support

`ClientProxyBase` is API-version aware via `ICurrentApiVersionInfo` (`Volo/Abp/Http/Client/ClientProxying/ICurrentApiVersionInfo.cs`):

```csharp
public interface ICurrentApiVersionInfo
{
    ApiVersionInfo? ApiVersionInfo { get; }
    IDisposable Change(ApiVersionInfo? apiVersionInfo);
}
```

The default implementation `CurrentApiVersionInfo` (singleton) stores the active `ApiVersionInfo` in an `AsyncLocal<ApiVersionInfo?>`. `ApiVersionInfo` itself records both the version string and the binding source (`Path`, `Query`, etc.), and exposes `ShouldSendInQueryString()` — when the action's metadata picks a non-path binding the version is appended to the query string by `ClientProxyUrlBuilder`. See [API Versioning](/http/api-versioning) for the abstraction side.

## `IClientProxyApiDescriptionFinder`: static vs dynamic

There are two finder interfaces with confusingly similar names:

- **`IApiDescriptionFinder`** (`DynamicProxying`) — the network-driven finder used by `DynamicHttpProxyInterceptor`. Reads `/api/abp/api-definition` at runtime.
- **`IClientProxyApiDescriptionFinder`** (`ClientProxying`) — used by *static* proxies. It walks an in-memory cache (`ClientProxyApiDescriptionFinder`) populated from generated metadata files at startup.

Both produce an `ActionApiDescriptionModel`, but the static path is faster because the metadata is embedded in the client assembly. `ClientProxyBase` only depends on the static finder, while the dynamic interceptor uses the dynamic finder to *resolve* the action and then hands the resulting `ClientProxyRequestContext` to the base for actual HTTP work.

## `ApiDescriptionCache`

`Volo/Abp/Http/Client/DynamicProxying/ApiDescriptionCache.cs` is a thin `ConcurrentDictionary<string, ApplicationApiDescriptionModel>` keyed by base URL. Because the entries are cached per base URL, a client process talking to multiple ABP backends (or a multi-tenant setup whose base URL contains a `{tenant}` placeholder) ends up with one cache entry per resolved URL. The default cache has no expiration — if you redeploy a backend with schema changes you must restart the client process.

<Note>
The class is intentionally minimal: there is no `IDistributedCache` integration. If you need cross-process invalidation you can replace `IApiDescriptionCache` in DI with your own implementation; the interface is `Task<ApplicationApiDescriptionModel> GetAsync(string baseUrl, Func<Task<ApplicationApiDescriptionModel>> factory)`.
</Note>

## When to choose dynamic over static proxies

| Concern | Dynamic | Static |
| --- | --- | --- |
| First-call latency | One extra HTTP round trip per base URL | Zero |
| Memory footprint | Holds full `ApplicationApiDescriptionModel` in memory | Holds only the action models you use |
| Recompile required when server adds endpoints | No | Yes — re-run `abp generate-proxy` |
| Reflection at startup | Minimal (interceptor wiring) | None (concrete classes) |
| Best fit | Internal microservices where contract and host ship together | Public SDKs, mobile clients, AOT-style scenarios |

`AddHttpClientProxies` registers dynamic proxies; `AddStaticHttpClientProxies` (same file) writes the same dictionary but skips the Castle interception layer because the generated proxy classes inherit `ClientProxyBase<TService>` directly. Both still use `ClientProxyBase` for the actual transport.

## Disposing and lifetime

Every piece in the dynamic stack is registered transient: `DynamicHttpProxyInterceptor<T>` is `ITransientDependency`, the Castle proxy is `services.AddTransient(type, …)`, and `HttpClientProxy<T>` is registered with the transient lifetime. The underlying `HttpClient` instances come from `IHttpClientFactory` so their handler lifetimes are managed by ASP.NET Core's `DefaultHttpClientFactory` — pooled and rotated every two minutes by default.

For a complete map of the surrounding HTTP stack, including how the server side exposes `/api/abp/api-definition` and how shared options like `AbpRemoteServiceOptions` are populated, see the overview page of this group.
